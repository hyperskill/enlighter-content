The Knowledge Graph Memory MCP is a tool that provides persistent memory capabilities using a local knowledge graph. This powerful server enables Cursor to remember information about users and projects across chat sessions, creating a more personalized and contextually aware experience. This stage will guide you through setting up and utilizing this MCP in your Cursor workflow.

Features:
- Store information in a structured knowledge graph
- Create and manage entities with unique identifiers and types
- Define relationships between entities
- Add and manage observations (discrete pieces of information)
- Search and retrieve stored knowledge
- Maintain context across multiple chat sessions

Key Use Cases
- Personalized user interactions
- Long-term project context maintenance
- Storing coding standards and preferences
- Building comprehensive knowledge of project requirements
- Tracking relationships between project components

Setup Instructions

You can set up the Knowledge Graph Memory MCP using either NPX or Docker. Choose the method that best suits your environment:

NPX Setup
Create or edit a file .cursor/mcp.json to include the following MCP setup: { "mcpServers": { "memory": { "command": "npx", "args": [ "-y", "@modelcontextprotocol/server-memory" ] } } }
Docker Setup
Create or edit a file .cursor/mcp.json to include the following MCP setup: { "mcpServers": { "memory": { "command": "docker", "args": [ "run", "-i", "-v", "claude-memory:/app/dist", "--rm", "mcp/memory" ] } } }

You can also customize the storage location using environment variables:

Create or edit a file .cursor/mcp.json to include the following MCP setup: { "mcpServers": { "memory": { "command": "npx", "args": [ "-y", "@modelcontextprotocol/server-memory" ], "env": { "MEMORY_FILE_PATH": "/path/to/custom/memory.json" } } } }

Core Knowledge Graph Concepts

The Knowledge Graph Memory MCP organizes information using three primary components:

Entities
- Primary nodes in the knowledge graph
- Each has a unique name (identifier)
- Defined by an entity type (e.g., "person", "organization", "project")
- Contains a list of observations (discrete pieces of information)
Relations
- Directed connections between entities
- Always stored in active voice
- Describe how entities interact or relate to each other
- Consist of a source entity, target entity, and relation type
Observations
- Discrete pieces of information about an entity
- Stored as strings attached to specific entities
- Should be atomic (one fact per observation)
- Can be added or removed independently
