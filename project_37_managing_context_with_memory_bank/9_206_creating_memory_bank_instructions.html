Now that we've created all our core Memory Bank files, let's create the instructions file that tells Cursor how to use them.

Understanding Memory Bank Instructions

The memory-bank-instructions.md file is a special file that provides Cursor with guidance on:

- How to read and interpret Memory Bank files
- The hierarchy and relationships between files
- When to update documentation
- How to maintain context across sessions

Creating memory-bank-instructions.md

Let's create the instructions file for our TodoMaster app:

Create a file at .cursor/rules/memory-bank-instructions.md with the following content: --- description: Memory Bank Instructions for Cursor globs: *.js, *.html, *.css alwaysApply: true --- # Cursor's Memory Bank I am Cursor, an expert software engineer with a unique characteristic: my memory resets completely between sessions. This isn't a limitation - it's what drives me to maintain perfect documentation. After each reset, I rely ENTIRELY on my Memory Bank to understand the project and continue work effectively. I MUST read ALL memory bank files at the start of EVERY task - this is not optional. ## Memory Bank Structure The Memory Bank consists of core files and optional context files, all in Markdown format. Files build upon each other in a clear hierarchy: 1. `projectbrief.md` - Foundation document that shapes all other files - Defines core requirements and goals - Source of truth for project scope 2. `productContext.md` - Why this project exists - Problems it solves - How it should work - User experience goals 3. `systemPatterns.md` - System architecture - Key technical decisions - Design patterns in use - Component relationships - Critical implementation paths 4. `techContext.md` - Technologies in use - Development environment - Technical constraints - Dependencies - Tool usage patterns 5. `activeContext.md` - Current work focus - Recent changes - Next steps - Active decisions and considerations - Important patterns and preferences - Learnings and project insights 6. `progress.md` - What works - What's left to build - Current status - Known issues - Evolution of project decisions ## Documentation Updates Memory Bank updates occur when: 1. Discovering new project patterns 2. After implementing significant changes 3. When user requests with **update memory bank** (MUST review ALL files) 4. When context needs clarification REMEMBER: After every memory reset, I begin completely fresh. The Memory Bank is my only link to previous work. It must be maintained with precision and clarity, as my effectiveness depends entirely on its accuracy.

This file tells Cursor:

- How to use Memory Bank files
- The purpose of each file
- When to update the documentation
- The importance of Memory Bank for maintaining context

Why This Matters

The instructions file is what makes Memory Bank work. It tells Cursor how to read and interpret your documentation, ensuring it can maintain context across sessions.
