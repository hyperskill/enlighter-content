Now that we understand the Memory Bank concept, let's start creating our files. We'll begin with the most important file: projectbrief.md.

Setting Up the Rules Directory

First, let's create a directory for our Memory Bank rules:

Create a directory structure for our Memory Bank rules: mkdir -p .cursor/rules/memory-bank Creating projectbrief.md

Now, let's create the projectbrief.md file for our TodoMaster app:

Create a file at .cursor/rules/memory-bank/projectbrief.md with the following content: --- description: Project Brief for TodoMaster globs: *.js, *.html, *.css alwaysApply: true --- # Project Brief: TodoMaster ## Overview TodoMaster is a simple web application that helps users organize and track their tasks. ## Core Requirements - Create, edit, and delete tasks - Organize tasks into categories - Set due dates and priorities - Mark tasks as complete ## Goals - Provide a clean, intuitive interface - Make task management quick and efficient - Help users stay organized ## Scope - Web-based application - Single-user focus - Local storage for data persistence - NOT included: user accounts, sharing features ## Success Criteria - Users can manage tasks in under 3 clicks - All data persists between sessions - Interface works on both desktop and mobile

This file establishes the foundation of our TodoMaster project. It defines:

- What we're building (a task management app)
- Core features (task CRUD, categories, priorities)
- Goals (clean interface, efficiency)
- Scope (what's included and what's not)
- Success criteria (how we'll know it's working well)

Understanding the Rule Format

Notice the metadata at the top of the file:

``` --- description: Project Brief for TodoMaster globs: *.js, *.html, *.css alwaysApply: true --- ```

This tells Cursor:

- What this rule is for (description)
- Which files it applies to (globs)
- That it should always be applied (alwaysApply)
