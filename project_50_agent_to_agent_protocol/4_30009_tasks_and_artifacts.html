How it Works: Tasks and Artifacts

Once a client agent has read another's Agent Card and knows what skills it offers, it can start assigning work. This is done through Tasks and Artifacts.

  • Tasks: The Unit of Work
    A "task" is the fundamental unit of work in the A2A protocol. When one agent wants another to do something, it sends it a task. Every task has a clear lifecycle, moving through states like submitted, working, completed, or failed. This design is robust, supporting both quick, simple requests that can be completed in seconds (synchronous tasks) and more complex, long-running jobs that might take minutes or hours (asynchronous tasks).
  • Artifacts: The Result of the Work
    When a task is successfully completed, its final output is called an "artifact." An artifact is the tangible result that the client agent receives. If the task was to suggest_chord_progressions, the artifact would be a structured JSON object containing the list of chords. If the task was to generate_logo_image, the artifact would be the image file itself.
Key Design Principles

The A2A protocol was built with a few core principles in mind to ensure it fosters a healthy and powerful ecosystem:

  • Open and Interoperable: A2A is an open standard, not a proprietary product. This is crucial because it ensures anyone can build a compatible agent, preventing vendor lock-in and encouraging a diverse marketplace of specialized AIs.
  • Secure by Design: The protocol was designed for professional, enterprise-grade use from day one. It supports standard authentication and authorization schemes to ensure that agents can communicate with each other securely.
  • Modality Agnostic: Communication is not limited to just text. The protocol is designed to handle a wide variety of data types, or "modalities." Agents can exchange structured data, files, images, and other media, making it a versatile tool for a broad range of applications.
