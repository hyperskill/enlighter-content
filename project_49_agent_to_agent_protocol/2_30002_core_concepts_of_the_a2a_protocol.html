What is A2A at its Core?

At its heart, the A2A protocol is a set of rules for communication. It is not a new programming language or a complex framework you need to learn from scratch. Instead, it's a specification that leverages familiar, time-tested web standards like HTTP and JSON. This makes it accessible and relatively straightforward to implement.

A useful analogy is to think of A2A as the universal postal service for AI agents. The postal service doesn't care if you're sending a handwritten letter, a legal document, or a birthday gift. It only cares that the package has a clear address, a recipient, and follows the rules for postage and handling. Similarly, A2A defines a standard way for a "client" agent to send a "task" (the package) to a "remote" agent (the recipient) and get a result back. It provides the "addressing" and "handling" rules so that the agents don't have to worry about the low-level details of their interaction.

A2A and MCP: Complementary Protocols

It's important to clarify the relationship between the Agent-to-Agent (A2A) protocol and the Model Context Protocol (MCP). They are not competing standards; they are complementary technologies designed for different, but related, purposes.

A2A is the protocol for inter-agent communication—how one agent assigns tasks to and gets results from another. MCP, on the other hand, is a protocol for intra-agent operations—how a single agent accesses its own private knowledge, tools, and capabilities. Each agent in an A2A network can have its own unique MCP server that provides it with specialized context or allows it to perform specific actions.

For example, imagine a "Research Agent" that is part of an A2A network. A "Writer Agent" could send it a task via A2A to find information on a specific topic. The Research Agent, upon receiving this task, might then use its own internal MCP server to access a set of tools for searching the web, querying academic databases, or summarizing articles. Once it has gathered and processed the information using its MCP-powered tools, it would package the result into an artifact and send it back to the Writer Agent using the A2A protocol. In this way, A2A and MCP work together to create a powerful, modular, and scalable system of AI collaboration.

The Core Components: Agent Cards and Skills

For one agent to be able to talk to another, it first needs to know who the other agent is and what it can do. A2A solves this with two simple but powerful concepts: Agent Cards and Skills.

  • The Agent Card: An AI's Business Card
    The Agent Card is a simple JSON file that acts as a public profile or "business card" for an agent. It's how an agent introduces itself to the world. Other agents can read this card to discover it and understand its capabilities. A typical Agent Card contains essential information like the agent's name, a description of its purpose, the URL where it can be contacted (its endpoint), and the security methods it requires for communication.
  • Skills: An Agent's Capabilities
    A "skill" is a specific, well-defined capability that an agent offers. Skills are listed within the Agent Card, forming a menu of services that other agents can request. For example, a "Music Theory Agent" might have skills like identify_key_signature or suggest_chord_progressions. Defining capabilities as discrete skills makes it very clear what an agent can and cannot do.
How it Works: Tasks and Artifacts

Once a client agent has read another's Agent Card and knows what skills it offers, it can start assigning work. This is done through Tasks and Artifacts.

  • Tasks: The Unit of Work
    A "task" is the fundamental unit of work in the A2A protocol. When one agent wants another to do something, it sends it a task. Every task has a clear lifecycle, moving through states like submitted, working, completed, or failed. This design is robust, supporting both quick, simple requests that can be completed in seconds (synchronous tasks) and more complex, long-running jobs that might take minutes or hours (asynchronous tasks).
  • Artifacts: The Result of the Work
    When a task is successfully completed, its final output is called an "artifact." An artifact is the tangible result that the client agent receives. If the task was to suggest_chord_progressions, the artifact would be a structured JSON object containing the list of chords. If the task was to generate_logo_image, the artifact would be the image file itself.
Key Design Principles

The A2A protocol was built with a few core principles in mind to ensure it fosters a healthy and powerful ecosystem:

  • Open and Interoperable: A2A is an open standard, not a proprietary product. This is crucial because it ensures anyone can build a compatible agent, preventing vendor lock-in and encouraging a diverse marketplace of specialized AIs.
  • Secure by Design: The protocol was designed for professional, enterprise-grade use from day one. It supports standard authentication and authorization schemes to ensure that agents can communicate with each other securely.
  • Modality Agnostic: Communication is not limited to just text. The protocol is designed to handle a wide variety of data types, or "modalities." Agents can exchange structured data, files, images, and other media, making it a versatile tool for a broad range of applications.
