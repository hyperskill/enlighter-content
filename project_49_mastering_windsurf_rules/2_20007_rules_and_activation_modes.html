Now that you understand why rules are useful, let's dive into how they work. In this section, we'll cover the different scopes for rules, the various ways they can be activated, and walk through creating your first global and workspace-specific rules.

Global vs. Workspace Rules

As we covered briefly, rules can exist in two scopes. The distinction is crucial for managing your personal preferences versus project-wide standards.

  • Global Rules (global_rules.md): These are your personal, universal guidelines that apply across all your projects. They are perfect for defining your preferred coding style or general instructions you always want Cascade to follow, regardless of the repository you're in. You manage these directly through the Windsurf UI.

  • Workspace Rules (.windsurf/rules/): These rules live inside a specific project's repository. They are designed for project-specific instructions, such as adhering to a team's style guide, using certain libraries, or understanding the project's architecture. Because they are part of the repository, they can be version-controlled and shared with your entire team.

Creating Your First Global Rule

The easiest way to get started is through the Cascade interface. Find the Customizations icon in the top-right corner menu (highlighted in the screenshot below).

Select the Rules tab and you will see the buttons to create global and workspace rules.

Click on + Global button and it will open the global_rules.md file in the editor. You can simply start typing to create your first rule.

A good example of a global rule is a guide for your coding style that you want to enforce across all your projects. You can use this to enforce your preferred coding style, such as using f-strings for string formatting, type hinting all function definitions, and keeping lines under 90 characters.

# My Python Coding Guidelines - Use f-strings for string formatting. - Type hint all function definitions. - Keep lines under 90 characters. Be careful with adding too many global rules. It can cause Cascade to get confused, uses extra context tokens, and can decrease the overall performance.

Well done, you have created your first global rule. Now let's create a workspace rule for a specific project.

Creating Your First Workspace Rule

To create a workspace rule, you need to create a new rule file in the .windsurf/rules/ directory. You can do this by clicking the + Workspace button in the Rules tab as described in the previous section.

Let's add the following rule to the .windsurf/rules/js-docs.md file:

# JavaScript Documentation Standard - All public functions must have JSDoc comments. - Include a `@description` for the function's purpose. - Use `@param {type} name` for all parameters. - Use `@returns {type}` to describe the return value.

At the top of the file, you will see the activation mode for the rule. Let's dive deeper into what these activation modes are and how to use them.

Understanding Activation Modes

Not every rule needs to be active all the time. Windsurf gives you precise control over when a rule should be applied using four distinct activation modes.

  • Always On: The most straightforward mode. The rule is always active and will be applied by Cascade in every relevant context.

  • Manual: The rule is inactive by default but can be manually triggered by typing @ in the chat input and selecting the rule by name. This is useful for on-demand instructions.

  • Model Decision: You provide a natural language description of when the rule should be used, and the AI model decides whether to apply it based on the current conversation's context.

  • Glob: The rule is automatically applied to any file that matches a specific file pattern you define (e.g., *.ts or src/components/**/*.jsx). This is extremely powerful for enforcing language- or framework-specific standards.

You can set this rule to Always On for now. In the next section, we will explore how to get the most out of different activation modes.
